# ADR: Task 生命周期边界

## 状态

已接受

## 背景

当前项目的聊天主链路基于 SSE 流式响应。前端需要看到任务开始、步骤推进、内容流式返回和完成状态，但并没有“后台异步任务中心”这类需求。

Worker 运行在 Cloudflare Workers 上，天然是多实例、短生命周期、无共享内存的执行环境。如果在当前阶段引入持久化任务中心，会显著增加实现复杂度，包括 Durable Object 协调、跨实例查询、一致性设计和清理策略。

## 决策

当前将 Task 状态定义为 **Worker 实例内实时态**：

- Task 只保证在单次请求生命周期内可见
- 前端通过 SSE 订阅 Task/Step 事件获取实时状态
- `/stats` 仅作为当前实例的近似观测接口
- 当前不支持持久化任务查询、任务恢复或跨实例统计一致性

## 影响

优点：

- 实现简单，和当前 SSE 交互模型一致
- 没有过早引入持久化协调层
- 便于优先完成聊天体验、技能路由和文件处理能力

代价：

- 不能把 Task 当成长期可查询资源
- `/stats` 不能用于精确运维指标
- 后续如果要做任务重放、审计或恢复，需要引入额外存储层

## 后续演进条件

只有在出现以下真实需求时，才考虑将 Task 状态迁移到 Durable Object、KV 或数据库：

- 用户需要查看历史任务执行状态
- 需要跨实例聚合任务统计
- 需要失败任务重试或恢复
- 需要对工具调用链路做审计与回放
